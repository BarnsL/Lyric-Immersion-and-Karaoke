## What the app is

Public repo **`BarnsL/Desktop-Karaoke`**
(https://github.com/BarnsL/Desktop-Karaoke). A transparent, always-on-top,
click-through Windows overlay (tkinter) that floats **synced furigana + romaji +
English translation** over Spotify / YouTube / any player, for language learning.
Reads real playback position from Windows GSMTC; identifies the song by **sound**
(Shazam) when titles lie. Languages: JA, ZH, KO, ES, DE, RU.

Portable **onedir** app: `_DATA` = the folder next to the `.exe` (holds `lyrics/`,
`settings.json`, `karaoke.log`).

## Build / run notes (IMPORTANT)

- **Use a full CPython with the deps + PyInstaller** — not a stripped-down or
  embedded venv. On a typical Windows per-user install that's:
  ```
  %LOCALAPPDATA%\Programs\Python\Python312\python.exe
  ```
  Some shells put a *different*, dependency-light Python first on `PATH` (an
  embedded tool venv with no `pip` / no `syncedlyrics` / `fugashi` / PyInstaller),
  which makes builds and dep-heavy tests fail with `No module named …`. Call the
  full interpreter **explicitly**. For Japanese text on stdout in a shell, set
  `PYTHONIOENCODING=utf-8`.
- **Build is ONEDIR** (`DesktopKaraoke.spec`, `exclude_binaries=True` + COLLECT).
  Onefile was rejected (re-extracts the ~50 MB UniDic dictionary on every launch).
  ```
  python -m PyInstaller --noconfirm --clean DesktopKaraoke.spec
  ```
  ~3 min → `dist\DesktopKaraoke\DesktopKaraoke.exe` (+ `_internal\`).
- **COLLECT fails with `WinError 145 directory not empty` if `dist` is locked.**
  Before rebuilding: stop any running `DesktopKaraoke.exe`, then
  `Remove-Item -Recurse -Force dist\DesktopKaraoke, build\DesktopKaraoke` (retry a
  couple times — AV/open handles can hold it briefly).
- **Rebuilding WIPES `dist\DesktopKaraoke\lyrics`.** Re-seed after every build:
  `Copy-Item lyrics\*.json dist\DesktopKaraoke\lyrics\ -Force`.
- Local modules are pinned in the spec `hiddenimports` (`songchange`, `api`,
  `character`, `recognize`, `fetch_lyrics`) since they're imported lazily.
- The app icon (`icon.ico`, generated by `make_icon.py`) is **embedded at build
  time**, so changing it needs a rebuild for the tray/taskbar/Start-menu to update.

## How the installed app is wired (Windows)

- **Start-Menu** shortcut → `dist\DesktopKaraoke\DesktopKaraoke.exe`. So
  rebuild + re-seed + relaunch the EXE = update the installed app.
- **Startup** (run-at-boot) shortcut → `pythonw.exe main.py` (runs the **source**,
  so it's current whenever the source is). Either entry point can be the live
  overlay.
- When updating the live app, identify the overlay specifically by the
  `DesktopKaraoke.exe` process (or `pythonw main.py`) — **don't blanket-kill
  `python.exe`**; unrelated local tools may also be running Python.

`http://127.0.0.1:8765`, localhost-only, every response `{"ok": …}`, hardened

- `POST /identify` `/wrong` `/nudge?s=2.5` `/reset` `/reindex`
- `karaoke.log` (rotating, in `_DATA`) records every match/sound/swap/sync decision.

## Work done in the 2026-06-22 session

### 1. Hardened the local API + security
`api.py`: total try/except wrapping, consistent `{ok}` shape, status codes,
`GET /health`, `GET /` schema, `POST /reset`, clamped `/nudge`, CORS, HTTP

secrets read from env only.

### 2. Seamless song-switching in compilations — `songchange.py` (NEW)
**Problem:** a long video with many songs back-to-back ("openings 1-26", album
upload, DJ set) keeps ONE GSMTC title, so song flips were caught only by the blind
~4 s Shazam poll → laggy swaps + constant CPU/network.
**Fix:** `SongChangeDetector` (daemon thread) = cheap RMS loudness meter on the
WASAPI loopback that detects the *music → near-silent gap → music* shape of a
track boundary and fires a callback → immediate re-identify.
- Wiring in `main.py`: `_start_boundary` / `_fire_boundary` (marshals to the Tk
  thread) / `_on_boundary` (throttled: skips if identifying or a boundary fired
  <4 s ago, sets `_fast_calib`, pulls `_start_identify` + `_arm_recal(6)`).
  `set_boundary` toggle persisted as `boundary` (default True). `quit()` stops the
  thread. Tray item "Fast song-change detect (compilations)". `/status` exposes
  `boundary_detect`.
- **Perf win:** `_recalibrate_loop` relaxes the blind Shazam poll to a slow
  heartbeat (`max(recal_secs, 25)`) once a song is **confirmed AND boundary on** —
  event-driven detection = far fewer Shazam calls over a long compilation.
- Conservative thresholds (absolute + relative silence floor, min-gap, a
  was-preceded-by-music latch, post-fire debounce) so quiet passages don't
  false-fire. Crossfaded comps with no gap fall back to the heartbeat.
- **Two bugs found + fixed while testing:** (a) `music_for` was zeroed on the first
  silent block so the "preceded by music" check always failed → switched to a
  latched `had_music` flag; (b) the stop `Event` was named `self._stop`, which
  **shadows `threading.Thread._stop`** and breaks `is_alive()`/join → renamed
  `self._stop_evt`. Inline regression tests cover both.

### 3. New app icon — `make_icon.py` (NEW) + `icon.ico` (replaced)

### 5. Romaji → original-script lyric fix — `fetch_lyrics.py`
**Problem:** a Japanese song could show **romaji only, no kanji, no translation**
(e.g. *Into Starlight* / IA). Cause: many lyric sites host a **romaji upload** of
Japanese tracks; with an English title the language gate didn't fire, the romaji
hit matched first, and `lang` came out `other` → no furigana, no translation.
**NetEase has the full kanji/kana original.**
**Fix:**
- `_looks_romaji(text)` — detects romanized Japanese (mora-shaped words +
  unmistakable JP tokens in `_ROMAJI_MARK`; gated by `detect_lang(text)=='other'`
  so vowel-rich Spanish/Italian aren't misread). Unit-tested.
- `_synced_cjk(title, artist, duration)` — searches syncedlyrics NetEase-first and
  returns the first synced result that actually contains CJK and verifies.
- `fetch_lrc` restructured: an inner `take()` stashes a romaji-only hit (instead of
  returning it) and keeps searching; after LRCLIB, if only romaji was found it
  calls `_synced_cjk` to **upgrade to the original**; the romaji is used only if no
  original exists anywhere.
- New `ja-romaji` language tag (`_song_lang`) still routes a romaji fallback
  through translation (`_translate_lines` `whole` set + `main.py _maybe_translate`),
  so the worst case is romaji + English, never romaji alone.
- **Verified live:** *Into Starlight* now resolves to the NetEase Japanese — 43/43
  lines with kanji+furigana, romaji, English. A library scan found one more
  romaji-only file (*unravel*) → re-fetched & upgraded too.
- **Sync note:** the "very off" timing was largely the wrong (romaji) LRC's
  timestamps; the NetEase original has correct timing. Shazam calibration + the
  boundary detector handle residual; any MV-vs-single offset = tray "Sync timing".

## Known limits / possible next steps

- Crossfaded compilations (no inter-track gap) won't trip the boundary detector →
  the slow Shazam heartbeat is the backstop. A spectral-novelty detector would
  catch them but costs CPU (deferred; see RESEARCH §3b).
- Word-level karaoke timing isn't available from free providers (RESEARCH §1).
- Truly niche VTuber/indie tracks may have no lyrics anywhere → `add_lrc.py`.
- Musixmatch intermittently 401s upstream inside syncedlyrics (NetEase + LRCLIB
  carry the load).
- tkinter Canvas is CPU-bound (no GPU); a Qt/Direct2D rewrite is the only
  step-change for raw render performance (deferred).

## Work done in the 2026-06-22 session (part 2): Store packaging + auto-updater

Goal: make this a **true one-click install for a non-technical user**. Diagnosis:
the app code was fine — the blocker is Windows trust. **Smart App Control blocks
unsigned `.exe`/installers outright**, and SmartScreen warns on them; there is no
free way to get that trust. Chosen path: **Microsoft Store (MSIX)** — the Store
signs the package with a Microsoft-trusted identity, so it installs with no
warnings and auto-updates. (See `STORE_SUBMISSION.md`.)

### New files
- **`appdata.py`** — single source of truth for the writable data dir. **Behaviour
  change:** portable `.exe` still uses next-to-exe, but a **packaged (MSIX) install
  uses `%LOCALAPPDATA%\DesktopKaraoke`** because the MSIX install dir is read-only.
  `main.py` (`_DATA`) and `fetch_lyrics.py` (`LYRICS_DIR`) both call
  `appdata.data_dir()` so they always agree. `is_packaged()` uses
  `GetCurrentPackageFullName`.
- **`version.py`** — `__version__` single source; `api.py` `/health` and
  `updater.py` read it. Bump it with the MSIX `Version` when releasing.
- **`updater.py`** — in-app updater. **Store/MSIX:** no-op (`check()` returns None;
  Store updates it). **Portable `.exe`:** checks GitHub Releases; if a newer build
  is published as a **`.zip` of the onedir folder**, downloads it and launches a
  helper (`apply_update.cmd`) that waits for the app to exit, `robocopy /E` (no
  purge → lyric cache + settings survive), and relaunches. **Source:** no self-
  update. Wired into `main()`: tray item "Check for updates" / "⬆ Install update
  vX" + a 15 s-delayed background check that notifies via the tray.
- **`make_assets.py`** — renders the Store tiles/logos from `icon.ico` via
  `make_icon._compose` into an `Assets/` dir.
- **`packaging/build_msix.ps1`** + **`AppxManifest.template.xml`** — build a
  full-trust MSIX (`runFullTrust` + `internetClient`). Produces
  `dist\DesktopKaraoke.msix`. Pass `-CertThumbprint` for a signed local-test
  package; leave unsigned for Store upload (the Store re-signs).
- **`STORE_SUBMISSION.md`** — the Partner Center steps that can't be scripted.

## Work done in the 2026-06-22 session (part 3): matching accuracy + render perf

### 1. Song name was being stripped from 「」/『』 titles (`clean_title`, main.py)
JP uploads put the song INSIDE brackets: `Daoko「COMIT COMET」(with BEATBROS.) -
TVアニメ『よわよわ先生』OPテーマ`. The old `clean_title` *stripped* all bracketed text,
leaving the generic `Daoko - TVアニメOPテーマ`, which fetched a DIFFERENT Daoko song
and then false-confirmed (the sound-ID title was genericised to the same garbage).
Fix: **extract** the 「…」 song (or a non-tie-in 『…』, which catches `『水星』`) instead
of stripping it; `_is_generic_title()` stops a generic descriptor (テーマ…) from
overriding a real Shazam ID. Coexists with another agent's "Artist / Song -ver-"
MV-title handling (their `-ver-` strip + my bracket extract are both in `clean_title`).

### 2. Concerts were read as one song (`is_live_or_compilation`, main.py)
A video titled `【#FG3Dライブ】MAKE IT, BREAK IT` cleans to `MAKE IT, BREAK IT`, which IS
a real song — so it title-matched and showed that one song's lyrics for the whole
concert, and Shazam (can't fingerprint live arrangements) never corrected it.
`is_live_or_compilation(title, duration)` flags live/concert/fes/medley/3D-LIVE
keywords (JP+EN) or >12 min. In live mode `_on_track_change` **refuses the title**
(no title-match/fetch), shows "🎤 Live set — listening…", and lets SOUND drive every
track; `_recalibrate_loop` keeps re-listening fast (live songs often segue with no
silent gap for the boundary detector), and the concert title can't override Shazam
(`_consume_async` CJK-preference guard `... and not self._live_mode`). Live captures
use a longer clip (≥8s) for better recognition.

### 3. Scroll-through performance — READ THIS BEFORE TOUCHING THE RENDERER
**The hard wall: a Tk `PhotoImage` create/paste of a wide scroll-block image costs
~50–60ms on this machine** (`paint` itself is only ~11ms — the cost is the Tk image
transfer, measured with a standalone bench). So each block SPAWN (`_render_img_block`)
and each fill REPAINT (`_paint_block_img`→`photo.paste`) is ~50ms = ~3× the 16ms (60fps)
frame budget, and the fill repainted on **every sung character**. That's the stutter.

**DO NOT "fix" this by switching to text-item blocks (`_use_img=False`).** Measured:
spawning one text block (~610 `create_text` items, 5/char outline) is ~260ms, and
`cv.move` of a full stream (~7k text items) is **~480ms/frame (2fps)** because Tk
re-rasterizes every glyph on each move. Images are pre-rasterized → cheap to move;
text is catastrophic here. `_use_img` stays **True**.

What actually helped (commit "Smooth out scrolling…"):
- The belt `cv.move("strm", …)` stays **every frame** (smooth motion is cheap).
- The expensive part (visibility scan + spawn/despawn + fill) runs on a frame
  throttle (`_fill_skip`) AND is budgeted: `_spawn_budget=1` (spread the ~50ms
  spawns; blocks appear 1200px off-screen so latency is invisible) and the fill
  repaints at `_repaint_budget`/frame with a per-block `_fill_interval` (~5fps — a
  coarse karaoke sweep reads fine; per-char pastes were the killer).
- `_tick` caches `clean_title()` (was re-running ~5 regexes every frame).
- **fps meter:** `/status` now reports `render_fps` (+ `perf`, `fps_target`,
  `live_mode`). Use it to measure. NOTE it only updates while lyrics are actually
  rendering (playing + lines loaded); it reads blank between songs.

**Measuring perf reliably:** live YouTube readings are confounded by song content
(line length/density changes the PIL load 2×). Use a controlled bench instead:
instantiate `Overlay` with `api.start_api` and `_start_boundary` patched to no-ops,
patch `_on_track_change` to a no-op, `ov.load(<cached .json>)`, set `scroll_dir="rl"`,
feed a fake `media.get()` with an advancing `position`, wrap `_ticker_update` to count
calls, run `_tick()` + `mainloop()` for ~6s. Same 90-line song: **47 → 66 fps (+40%)**
old→new. (Real-world over video is lower for both — the layered window compositing
over the playing video is extra cost outside the app's control.)

**The real ceiling** is still tkinter Canvas + Tk image transfer. A genuinely smooth
60fps with a per-character fill needs a different renderer (Qt/QML, Direct2D, or a
GL/skia canvas) — see the long-standing note in "Known limits". The above makes the
*current* renderer hit/clear its target and bounds the frame-time spikes; it does not
remove the underlying ~50ms/PhotoImage wall.
